## 鉄則（すべてのタスクに適用）

### 1. 確認ポイントルール

以下のタイミングでは **必ず翔伍さんに確認を取り、承認を得てから次に進めてください**：

- **タスク分解した直後**（「この手順で進めます。よろしいですか？」）
- **ファイルを新規作成・上書きする前**（「このファイルを作成/上書きします。よろしいですか？」）
- **外部API・外部サービスに接続する前**（Google Sheets API、freee、LINE等）
- **想定と違う状況に遭遇した時**（データ形式が違う、ファイルが見つからない等）
- **各ステップ完了時**（「Step 1 完了しました。結果は〇〇です。次に進めてよろしいですか？」）

確認なしで進めてよいのは：
- ファイルの読み取り（readのみ）
- ログ・進捗ファイルへの書き込み
- 情報の整理・分析（出力を伴わない思考作業）

※ このルールは翔伍さんからのフィードバックに応じて随時更新してください。

### 1.5. 伝達ボード更新ルール（2026-04-05 追加 / 2026-04-06 鉄則化）

**【鉄則】伝達ボードへの報告は義務です。怠った場合は方針違反とみなします。**

ダァイ（claude.ai）は伝達ボードからしか各エージェントの状況を把握できません。更新が遅れるとダァイが古い情報で翔伍さんに間違った提案をしてしまいます。これは翔伍さんに直接迷惑をかける行為です。

**必ず伝達ボード（board.md）に書き込むタイミング：**
- **毎ラリー（翔伍さんへの返答ごと）** ← 最重要。返答するたびに伝達ボードに現状を書き込むこと
- **作業開始時**（今日何をやるか）
- **ブロッカーが解消されたとき**
- **タスクが完了したとき**
- **新しいブロッカーが発生したとき**
- **1日の作業終了時**（その日の成果まとめ）

**「毎ラリー」とは：翔伍さんに何かを報告・提出・質問するたびに、同じ内容を伝達ボードにも書き込むということ。伝達ボードに書かなかった作業は「やっていない」のと同じ。**

**書き込みを忘れる＝ダァイの判断を狂わせる＝翔伍さんに迷惑をかける。絶対に忘れないこと。**

フォーマット（統一済み・厳守）：
```
- [（名前）→ダァイ] **現状報告** (時刻)
  - 完了: （完了した作業）
  - ブロッカー: （止まっている箇所。なければ「なし」）
  - 次: （次にやること1つ）
```

確認なしで書き込んでよい（伝達ボードへの報告書き込みは承認不要）。

### 1.6. 確認済み事項の記録ルール

**翔伍さんに確認して回答をもらったら、その内容をprogress.mdに即記録せよ。**
記録せずに次に進むことは禁止。

記録フォーマット：
```
- [確認済み] ○○について → 翔伍さん回答：「△△でOK」
```

**記録済みの事項を再度質問することは禁止。**
質問する前にprogress.mdを確認し、すでに回答があるならそれに従うこと。

翔伍さんに同じことを2回聞く＝翔伍さんの時間を奪う。方針違反です。

### 1.7. 着手前アウトプットルール（全エージェント必須）

**【鉄則】コードを書く前・作業する前に、必ず「完成イメージ」を翔伍さんに見せること。**

翔伍さんの頭の中にあるイメージとエージェントのアウトプットはすぐにズレる。
ズレたまま作り込むと全部やり直しになる。だから「作る前に見せる」。

例えるなら、家を建てる前に設計図を見せる。設計図OKが出てから建て始める。

**タスクを受けたら、実行する前に以下3点を出力せよ：**

1. **完成イメージ**
   - テキスト成果物 → 全文ドラフト or 構成案
   - コード → 処理の流れを日本語で書いた擬似コード
   - デザイン → モック画像 or ワイヤーフレーム
   - データ処理 → 入力データと出力データの具体例
   - ※翔伍さんが「これでOK」か「ここ違う」と判断できるレベルで出すこと

2. **やらないこと**（スコープ外を明示）
   - 「今回は○○はやりません」と先に伝える
   - 翔伍さんが「それもやって」と言えるようにするため

3. **不安な点**（ズレそうなポイント）
   - 「○○の部分は翔伍さんのイメージと違うかもしれません」
   - ズレそうな箇所を先に言うことで、手戻りを防ぐ

**翔伍さんから「OK」が出るまで実行に移らないこと。**
**「OK」が出たら、そのイメージ通りに作ること。途中で変えない。変えたくなったら再度確認。**

### 2. コンテキスト保持ルール

**あなたの記憶はセッションごとに消えます。必ずファイルに残してください。**

すべてのタスクで `progress.md` を作成・更新し続けてください：

```markdown
# 進捗ファイル（progress.md）

## タスク概要
（元のタスク指示をそのまま転記）

## 手順（タスク分解結果）
- [ ] Step 1: 〇〇
- [x] Step 2: △△ ← 完了
- [ ] Step 3: □□ ← 今ここ

## 現在の状況
- 今やっていること: Step 3 の □□
- 前のステップの結果: △△のファイルを作成済み（パス: ./output/xxx.py）
- 次にやること: □□を実行

## 判断ログ
- [2026-xx-xx] Step 2で〇〇という判断をしました。理由は△△です。
- [2026-xx-xx] 翔伍さん確認済み：□□の方針でOK。

## エラー履歴
（発生したエラーと対処を記録）
```

**タスク再開時は、まず `progress.md` を読んでください。**
読んでから「前回はStep Xまで完了しています。Step Yから再開してよろしいですか？」と確認してください。

### 3. エラー対処ルール

エラーが発生したら、以下の順序で対処してください：

1. **エラー内容を `progress.md` に記録する**
2. **原因を自分で分析する**（ログ、エラーメッセージ、ファイル状態を確認）
3. **自分で直せる場合** → 修正案を翔伍さんに提示 → 承認後に実行
4. **自分で直せない場合** → まず **伝達ボード経由でダァイに相談** → ダァイと解決策を検討 → それでも解決しなければ翔伍さんに報告

**翔伍さんへの報告ルール：**
- **「どうしたらいいですか？」「どうしましょう？」は禁止**
- 必ず **提案付き** で報告してください（「〇〇が起きました。A案またはB案で対処を考えています。」）
- 提案が却下されても諦めず、別の角度から新しい提案を出し続けてください

**以下の行動は禁止です：**
- 同じ方法で3回以上リトライすること
- エラーを無視して次に進むこと
- 「うまくいきませんでした」だけの報告（原因分析なしの報告は禁止）

### 4. 成果物ルール

- **成果物は「翔伍さんがクライアントにそのまま見せられるレベル」で出すこと。自分が「まあいいか」と思ったら、もう一段上げてから提出すること。**
- すべての成果物は `./output/` に保存してください
- コードを作成した場合は **必ず全文** を `./output/` に保存してください（差分・部分提供は禁止）
- 翔伍さんに確認してもらうコードは、チャットにも全文テキストで出力してください
- スクリーンショットの文字起こし・データのテキスト化は **コードブロック形式** で提供してください
- ファイルを作成したら、ファイル名・パス・内容の要約を報告してください

### 5. 作業開始の手順（毎回必ず実行）

タスクを受け取ったら、以下の順序で進めてください：

1. **伝達ボード（board.md）を確認** → ダァイからの指示・情報がないか確認
2. **自分のやらかしリスト（mistakes/）を確認** → 同じミスを絶対にしないため
3. `progress.md` が存在するか確認 → あれば読んで再開確認
4. なければ新規タスクとして開始
5. タスクを **最大5ステップ以内** に分解してください（細かくしすぎないこと）
6. 手順を翔伍さんに提示して承認を得る
7. 承認後、Step 1 から実行開始
8. 各ステップ完了ごとに `progress.md` を更新 → 翔伍さんに報告

### 6. 標準ワークフロー（全タスク共通）

**翔伍さんの一言で、以下の流れで進める。各ステップごとに翔伍さんの確認を取ること。**

```
翔伍さん：「○○やって」
    ↓
【Step 1】受け取り
  - タスクの内容を理解する
  - progress.mdに確認済み事項がないか確認（同じ質問を繰り返さない）
  → 翔伍さんに「この理解で合っていますか？」と確認
    ↓
【Step 2】完成イメージを見せる ← ★確認
  - 完成イメージ（ドラフト・モック・擬似コード等）
  - やらないこと
  - 不安な点
  → 翔伍さんの「OK」を待つ
    ↓
【Step 3】バックアップを取る ← ★確認
  - 既存ファイルを修正する場合、必ずバックアップを取ってから作業
  - バックアップパスをprogress.mdに記録
  → 「バックアップ取りました。作業に入ります」と報告
    ↓
【Step 4】作業する（小さく区切って進める）
  - 指示された箇所だけ触る。他は触らない
  - 確定版（翔伍さんOK済み）は絶対に壊さない
  - 翔伍さんの回答は即progress.mdに記録
  → 1つの修正が終わるごとに「ここまでできました」と途中経過を見せる ← ★確認
  → まとめて一気にやらない。小さく進めて都度確認
    ↓
【Step 5】チェックリストを通す ← ★確認
  - やらかしリストの項目を全てクリアしたか
  - クライアントにそのまま見せられるレベルか
  - 「まあいいか」と思ったら、もう一段上げる
  → 「チェックリスト全項目クリアしました」と報告
    ↓
【Step 6】Before→Afterで報告 ← ★確認
  - 修正前と修正後を並べて見せる
  - 「直しました」禁止。具体的な証拠で示す
  → 翔伍さんの「OK」を待つ
    ↓
【Step 7】完了
  - progress.mdを更新
  - 伝達ボードに報告
```

**全ステップで翔伍さんに確認を取ること。勝手に次に進まない。**
**特にStep 4は小さく区切って、途中経過をこまめに見せること。**
**翔伍さんが「まとめて進めていいよ」と言った場合のみ、まとめて進めてよい。**

### GASからAPIへの移行ルール

現在の処理はGAS（Google Apps Script）で動いていますが、**APIへ完全移行する方針** です。

移行は慎重に進めてください：
1. 既存のGASコードを全文読み込んで、何をやっているか完全に把握する
2. 同じ処理をAPI経由で再構築する
3. 両方を並行稼働させて、結果が一致するか検証する
4. 一致を確認できたら、翔伍さんの承認を得てからGAS側を停止する

例えるなら、ベテランパートさん（GAS）から新人（API）への引き継ぎです。いきなりベテランを辞めさせるのではなく、しばらく2人体制で同じ仕事をやらせて、新人が間違いなくできることを確認してから引き継ぎ完了です。

※ 既存のGASプロジェクトが複数存在して混乱が起きています。移行時にどのGASが何をやっているかを整理することも含めて進めてください。

- ソースシートは絶対に触らない。ミラーシートでも削除は危険
- 変更前にバックアップとして変更履歴のポイントを確認しておくこと

### 【商品構成を理解してから作業すること】
- マーケ講座とAI講座は別商品
- マーケ講座には流入経路が複数ある（オーガニック・広告経由・代理店・イベント等）
- ROIシートのタブ名がそのまま商品×経路の構造を反映している
- 「審査LINE 2,053件」のような大きな数字を見たとき、それが何の商品の何の経路かを必ず確認すること

### 【GASの危険な関数】
- setupAllTriggers()は全トリガーを削除するだけで新規作成しない
- 誰かが実行すると全自動処理が停止する
- doGet()にもトリガー全削除コードが入っていた（過去に発見済み）

### 【コスト最適化（高級シェフに皿洗いさせない）】
- 固定ロジック（データコピー、集計、転記）→ Pythonスクリプト（トークン消費ゼロ）
- 判断が必要な作業（OCR分類、条件判断、文章生成）→ Claude
- 最終確認・承認・方針決定 → 翔伍さん
- 「全部AIでやる」は禁止。VPSでClaude Codeを24時間回す必要もない

### 【データ検証の原則】
- 自分の集計結果とシートの表示値が違ったら、まず自分のロジックを疑う
- ソースの実データを直接見て突き合わせること
- 推測で進めず、実データで確認してから報告すること

### 【ダァイとの連携で学んだこと】
- ダァイはROIシートの実データをcurlで直接取得して分析できる
- ダァイがデータ検証→ゆうさんが実装、という役割分担が効率的
- 伝達ボード経由だけでなく、翔伍さんを通じて直接やり取りする方が早い場合もある
- ダァイからの指摘で気づけることが多い（Chatwork自動更新の見落とし、FB日別タブのNo.13〜15の見落とし等）
